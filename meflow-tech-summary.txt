总结

本次会议围绕 MF 3.0 技术架构部分展开培训与答疑，介绍了 MF 3.0 技术架构相较于 2.0 的升级内容、性能提升、数据安全等方面，还解答了客户二开难度、集成平台演示等问题，内容如下：
MF 3.0 技术架构介绍
架构图调整：赵世昌介绍，MF 3.0 架构图在开放平台原有架构图基础上做了调整和高亮。灰色部分与 2.0 基本一致，绿色部分是主服务和 meflow agent（原 copilot）的升级变化，主服务后端改 Java，前端改 react；黄色部分查成正在比对；红色部分可切换，自建服务推荐用 MySQL、Rabbitmq 和 Minio 的 S3 协议，关系型数据库可切换。

技术层面调整
前端后端技术变更：主服务前后端技术有大调整，后端改 Java，前端改 react，提升技术层面性能。
插件数据项与集成：插件数据项和与集成平台的集成部分有调整，相关内容在近期培训中大家已较了解。
各颜色部分含义
黄色部分情况：黄色部分前两天培训有提及，查成正在比对相关内容。
红色部分切换：红色部分可切换，如数据库可切换为 Oracle、达梦、Postgresql 等主流数据库，也包括达梦、人大金仓等。
MF 3.0 核心升级内容
性能升级
列表页数据处理：列表页数据完全基于 ES，查询和存储分离，增删改基于 Mongo，查基于 ES，天然支持全字段全文检索，轻松应对百万级数据。以合同台账页面为例，3.0 相比 2.0 性能提升显著，2.0 页面查询性能差，3.0 平均响应时间 0.21 秒，QPS 达 1128。

不同环境性能差异：陈莉娟提出 prod 环境加载不如演示丝滑，赵世昌解释演示主要是列表页展示，从头加载涉及前端页面资源重复加载、接口重复调用、资源分配不合理等问题，在技术需求优化范围内，具体页面投入精力可处理。
数据安全和隔离：2.0 是逻辑隔离，依赖研发人员自觉性；3.0 是物理隔离，数据库按租户独立部署，尤其针对 SaaS，对研发透明，除专门做数据隔离的研发人员外，平时写代码无需专门关注。
灵活扩展和定制
定制开发周期缩短：数据项插件、集成平台等减少定制开发周期，但因目前无项目从头到尾交付，需继续验证，且与 2.0 标品功能有差距，项目需完善标品功能。
版本升级影响可控：版本升级影响范围更可控，降低升级风险，如海天项目，标品出稳定版可及时更新到项目上，无需专门合并代码，只发版即可。
按需部署目标：目标是支持按需部署，底层已支持，但目前未完全拆开，预计 Q1、Q2 完成，2.0 基本不支持。
数据库兼容性：支持关联数据库切换，可直接换成 Oracle、达梦、Postgresql 等主流数据库，也包括达梦、人大金仓等。
智能化能力：上午已讲解，赵世昌不再细讲。
集成平台
内置集成平台优势：3.0 直接内置集成平台，有触发器、应用和集成流，降低系统集成技术门槛。如海天下项目的消息提醒、主数据对接、合同评审回传等都在集成平台完成。
集成平台功能示例：可实现自动化，如模板审批通过后更新模板状态，通过集成平台监听审批，调用接口完成。
常见问题解答
服务可用性问题：每多加一个 9 是指数级增长，且受客户自身网络或硬件条件限制，承诺过高可能造成误导。
性能提升数据：以李宁项目压测数据为例，2.0 数据量 1000 +，并发 100，QPS 25.55；3.0 平均响应时间 0.21 秒，QPS 1128。
技术栈相关
安全机制情况：3.0 数据权限、功能权限已做，基本防攻击机制符合期望，编辑支持大于 PS，基本编辑场景达标，但模板涉及书签的如留言定位等还在调研。
运维排查便利：3.0 从代码规范上强调添加链路，报错带 traceid，方便研发快速排查错误，也可借助大模型处理。
双因子情况：2.0 有双因子，3.0 目前没有，但开发成本不高，大鹏用户关注，可考虑添加。
审批引擎 Pod 问题：陈莉娟解答，此问题主要看售前产品能否根据客户业务场景在系统中做权限矩阵或配置验证，不涉及技术部署。
代码交付分级
各级交付内容：陈翱介绍，MF 3.0 代码交付分四级。一级交付依赖的 jar 包代码无法编译运行，用于代码安全扫描；二级交付包含依赖的 jar 包和定制代码，可编译运行，但不包含标品源代码；三级交付包含标品主模块源代码，需乐哥审批；四级交付涉及内部基础服务，如 Poweroffice、内部公共库、流程引擎、插件系统、BI 模块、Powerbase、集成平台、Meagent、GRPC 等，一般不交付。
特殊情况处理：若客户定制少，想对评审和审批的 3 - 5 个包做二开，可从三级交付中拆出作为定制后代码交付，不算三级交付。
其他要点
资源配置：目前 MF 3.0 资源配置与 2.0 保持一致，预留了一些资源。
数据安全：调用公网大模型时，数据安全受制于当前用户的功能权限和数据权限，可考虑做脱敏或过滤工作。
embedding 情况：目前检索用 ES，未引入向量数据库，后续可能引入，但优先级不高。
部署方式：支持容器化部署（如 K8S、C3S 集群）、Docker compose 部署，也可在客户物理机或虚拟机上部署。
审计日志：默认存储半年，可根据用户需求调整。
后续工作计划
模块化拆分：预计 Q1、Q2 完成 MF 3.0 按需部署的模块化拆分工作。
WPS 版本推出：预计 Q1 推出一版 WPS 版本，但部分功能可能受限，需看实际调研结果。
集成平台演示完善：若客户技术侧关注集成平台，申请技术同事参会交流，且建议自己上手试集成平台最简单的集成流并运行，结合培训视频演示效果最佳。

智能章节
02:31  3.0技术架构介绍及数据库、编辑器切换说明
本章节由赵世昌介绍3.0技术架构，此介绍面向售前和销售团队，文档多由AI生成。他展示架构图，指出相比2.0的改动，如主服务和meflow agent部分升级，后端改Java、前端改react。还介绍Mysql、Rabbitmq等用途及切换方式，提到编辑器可换WPS，部分功能待验证，预计Q1出有部分功能受限的WPS版本。
14:42  系统3.0对比2.0的核心升级及典型问题
本章节赵世昌介绍系统3.0相比于2.0的核心升级。包括性能升级，列表页数据基于ES，查询快；数据安全和隔离采用物理隔离；灵活扩展和定制可缩短开发周期、降低升级风险；支持数据库切换、按需部署；内置集成平台降低集成门槛。此外还提及一个典型问题“为何服务可用性不能达99.999%”及回答策略。
26:42  系统性能压测指标、场景及数据情况说明
本章节赵世昌介绍三点性能提升相关数据，以李宁项目任务列表页查询性能压测为例，提及数据量、并发、QPS等指标含义，说明并发是模拟用户同时访问情况，QPS是并发最高时每秒响应请求均值。还指出压测针对典型场景，未对每个场景压测以控成本，本地压测受网络带宽限制但数据已较好。
32:50  系统3.0技术栈及相关功能特性介绍
本章节赵世昌介绍技术栈相关内容，提及数据安全、权限控制，双因子 3.0 虽未实现但可先写入；3.0 在链路追踪、错误排查、审批流程、查询性能等方面有改进；移动端做了适配，支持 oauth 协议单点登录；消息集成中飞书已完成，钉钉和企微有问题待处理；数据支持导出导入，有 API 和 Excel 两种导入方式。
40:49  3.0日志情况、存储策略及服务资源配置说明
本章节赵世昌介绍日志情况，2.0维度日志分研发自用和日志审计两块，研发自用日志比2.0更多。Q25年Q4迭代的日志审计管理含操作动作、操作人等信息，默认存储半年，可调整，清理策略和保留时间视项目而定。目前用的组件开源无侵权风险，服务资源配置与2.0比总体无变化，稍预留资源。
43:28  服务器部署资源分配、高可用及适配减配说明
本章节赵世昌在陈莉娟的提议下，介绍服务器部署资源。提到资源主要围绕CPU、内存和磁盘考虑，标准配置总体是三台16、18和32G资源的服务器，测试环境可减配；高可用需增加副本数，服务部署两个节点以上且在不同机器；部分组件可换成客户自身的，相应资源会减配，还有组件版本要求和沟通模板。
47:41  系统部署方式、审计日志及专有名词说明
本章节由赵世昌介绍相关内容，提到部署方式主要涉及洪超运维方面，支持容器化部署，如对方有K8S或C3S集群可直接部署，也支持CON Docker compose部署，客户有物理机或虚拟机也能部署；审计日志会记录相关内容，默认存储半年，时长可按需调整；还在下面做了附录解释专有名词，若需进一步了解可借助AI查询。
49:11  客户二开难度及开发文档完善情况介绍
本章节赵世昌介绍了客户二开相关情况。相比2.0，客户二开难度大幅降低，可在不开放核心源码的情况下，对照维护于open doc的研发文档开发后端，文档随项目实施不断完善，如海天项目实施中就补充了缺失内容。除开发后端代码，部分功能还可借助平台做轻定制，如更改模板字段。
51:42  集成平台二开难度、亮点优势及演示策略
本章节赵世昌围绕集成平台展开介绍。提及可借助集成平台做轻度定制，关于演示策略，建议上手试最简单集成流并运行出结果、对照培训视频配置集成流。其亮点包括可自定义触发器，按设定时间或事件触发动作，能对外暴露接口拉数据、调用服务更新数据，核心是在不改动核心代码下完成数据集成等工作。
54:45  产品版本升级情况：轻定制可丝滑，重定制需评估
本章节围绕产品版本在客户处部署时的丝滑升级问题展开讨论。以海天项目为例，轻定制可丝滑升级，能直接拿标品最新代码发布；重定制若涉及模块复写，复写模块无法升级，耦合性高时需进一步评估，但升级成本低于2.0。一般情况下，如海天项目能直接私化升级，且升级时间由项目决定。
57:45  海天项目架构亮点、数据安全及审批POC探讨
本章节由赵世昌介绍海天项目相关情况，包括部分功能由标品完成后直接升级。阐述系统灵活和可配置性，如插件扩展、数据项、集成平台。提及资源配置与2.0保持一致，数据安全受制于用户权限，可考虑脱敏过滤。目前检索用ES，后续或引入向量数据库。还讨论了审批引擎做POC问题，由陈莉娟解答。
01:04:06  VFLOW 3源代码交付等级说明及相关安排
本章节主要讨论了源代码交付等级问题。石玏提出询问，赵世昌介绍了一级不交付、二级含依赖jar包和定制代码、三级含主模块源代码、四级为内部基础服务等交付等级情况。陈翱补充相关规则，强调三级交付需审批。此外，陈莉娟提醒大家了解3.0技术架构，售前遇集成平台问题可申请技术同事交流。
